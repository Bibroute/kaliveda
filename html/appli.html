update_runlist [data repository name] [dataset name] [data type]

Used to update the database of available data files in the data repository. Normally this database is kept automatically up to date, but this tool can be used in case things go wrong: files which can be seen to exist in the repository are not accessible from the KaliVeda analysis environment (KaliVedaGUI etc.).
Typing 'update_runlist' without arguments prints a reminder of the arguments to give.
[data repository name] : name of an accessible repository e.g. defined in user's .kvrootrc file. The repository must be LOCAL, we cannot check remote repositories.
[dataset name] : name of a known dataset, e.g. INDRA_camp1
[data type] : one of the known data types, e.g. raw, recon, ident or root


analyse_logs [jobname format] [list of log files]

Used to analyse the large number of BQS log files produced by "parallel" analysis of many runs at the IN2P3 Computing Centre. The result is a listing of which jobs were successfully completed, which were KILLED, etc., along with (for the successful jobs) information on the requested, minimum, mean and maximum CPU time, memory and scratch disk used, in order to help users to better tune their ressource requests to the needs of their analysis (users who systematically overestimate these requests risk punishment by the CCIN2P3!)
Typing 'analyse_logs' with no arguments prints a reminder of the arguments, and an example:

        Analysis of Grid Engine (CCIN2P3) batch log files

    analyse_logs [jobname]

    arguments are:
        [jobname] common root of name of all jobs to analyse
              e.g. "PbAu29MeV" for jobs with names like "PbAu29MeV_R8213" etc.
              The programme will attempt to read all files with names like
                  PbAu29MeV_R8213.o5647093
                  PbAu29MeV_R8214.o5647094
              etc. in the current directory


SetKaliVedaVersion [version name] [path to installation directories]

This tool has several uses:

1. it can be used to set up the directories needed for a new installation of KaliVeda, set the KVROOT environment variable, and modify the user's PATH and LD_LIBRARY_PATH accordingly
2. it can be used to change between different versions of KaliVeda on the same machine (automatically setting the correct KVROOT, PATH and LD_LIBRARY_PATH)
3. if used in the user's login script, it can automatically switch to the correct installation of KaliVeda depending on the local versions of ROOT and gcc on different machines (automatically setting the correct KVROOT, PATH and LD_LIBRARY_PATH)

Note: only works for Linux machines running the (t)csh shell
Note2: it is not installed when KaliVeda is installed; to use it, you must first copy the following files

[KaliVeda source directory]/tools/SetUpKaliVeda.csh
[KaliVeda source directory]/tools/SetUpKaliVedaDirectories.sh

into a directory which is in your PATH variable, e.g. $HOME/bin if it is in your PATH variable. Then create the following alias (csh syntax), which you should add to your .cshrc:

alias SetKaliVedaVersion 'source `which SetUpKaliVeda.csh`'
 

Case 1: creating directories for installation

After following the instructions in Note2 above, type one of the following commands:

SetKaliVedaVersion version /path/where/I/want/to/install
SetKaliVedaVersion version

replacing 'version' and '/path/where/I/want/to/install' by some real values! (If no path is given, your $HOME will be used by default, or the last path given if you have already used SetKaliVedaVersion).
This will create the following directories:

/path/where/I/want/to/install/.kvroot-version_name
/path/where/I/want/to/install/.kvroot-version_name_gcc-vx.x.x_root-x.x.x

with 'x.x.x' being replaced by the version numbers of gcc & ROOT respectively, and will set KVROOT to point to the second of the two directories (to save disk space, files which do not depend on the version of gcc or ROOT are installed in the first directory, and are accessible from the KVROOT directory through symbolic links). The PATH and LD_LIBRARY_PATH variables are also updated with, respectively, $KVROOT/bin and $KVROOT/lib being added to them.
You can now build & install KaliVeda by going to the top-level source directory and typing 'make' (see here for full description of Makefile targets & options).

If you want, you can set up the variables KVROOT, PATH and LD_LIBRARY_PATH the next time you login by simply typing the command

SetKaliVedaVersion

without any arguments. See Case 3 below for how to make this automatic.

Case 2 : using different versions of KaliVeda

If you install several versions of KaliVeda using the procedure described in Case 1, you can change from one version to another by typing the command

SetKaliVedaVersion new_version [/installation/path]

The installation path is only necessary if the new version is installed in a directory different from the current version.

Case 3 : automatically changing version when ROOT/gcc version changes

This option is very useful if, for example, you log into your account on several different machines, with different versions of ROOT and/or gcc. You can also use it quite simply to set KVROOT, PATH and LD_LIBRARY_PATH correctly when you log on to a single machine. Add the following line to your .cshrc (after the definition of the SetKaliVedaVersion alias):

SetKaliVedaVersion >& /dev/null

Now each time that you log in, your environment variables will be automatically set to use KaliVeda on the current machine. Note that if you log in to a machine with a version of ROOT or gcc which you have not encountered before, this will create the installation directories for a new build of KaliVeda, as in Case 1; all you have to do then is to perform the build. Note also that if you change versions of KaliVeda during a session (as in Case 2), this version will be used for all subsequent sessions (until you change again).
